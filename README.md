## System Overview

A Shakespeare question-answering system built on a pretrained Small Language Model
(**Qwen2.5-1.5B-Instruct**) with Retrieval-Augmented Generation (RAG). The system is
designed for users with no prior knowledge of Shakespeare and supports four interaction
types required by the assignment:

| Interaction type | Example query |
|---|---|
| Concept explanation | "Who is Hamlet?", "What is the role of Lady Macbeth?" |
| Contextual Q&A | "Why does Macbeth kill Duncan?", "Why does Hamlet delay?" |
| Evidence retrieval | Every answer displays retrieved source passages with full metadata |
| Stylised generation | Short responses (≤ 150 words) in Shakespearean style, clearly labelled |

Corpus: *Hamlet*, *Macbeth*, *Romeo and Juliet*

## Architecture

```
User query
    │
    ▼
EmbeddingRetriever                  ← sentence-transformers/all-MiniLM-L6-v2
    │  cosine similarity over
    │  pre-built embeddings.npy
    ▼
Top-8 speaker-turn chunks           ← each chunk: play / act / scene / speaker / text
    │
    ├─► Parent Document Retrieval   ← scene_lookup.json supplies full scene context
    │   (scene context injected      for each retrieved chunk's scene_id
    │    into prompt header)
    │
    ▼
Prompt builder (PDR format)         ← system_prompt.txt + scene context + ranked passages
    │
    ▼
Qwen2.5-1.5B-Instruct               ← local weights or HuggingFace auto-download
    │  greedy decoding, 300 new tokens
    ▼
Citation guard                      ← labels stylised responses as creative output
    │
    ▼
Answer + retrieved evidence         ← printed to terminal with play/act/scene/speaker
```

**Baseline comparison:** `baseline.py` runs the same model with a simpler prompt and
*no retrieval*. The evaluation script runs both systems on every question and writes a
CSV for manual scoring.

## System Components

### `config.py`
Central configuration. Sets `DATA_DIR`, `PROMPT_DIR`, `RESULTS_DIR`, `PLAY_FILES`,
`DEFAULT_TOP_K = 8`, and `EMBEDDING_MODEL_NAME = "sentence-transformers/all-MiniLM-L6-v2"`.
Change `DEFAULT_TOP_K` here to tune retrieval breadth globally.

### `data_loader.py`
Loads all three plays from their speaker-turn JSONL files and returns a flat list of
records. Each record carries: `play`, `act`, `scene`, `speaker`, `text`, `scene_summary`,
`keywords`.

### `chunking.py`
Converts records into chunk dicts. Key design decisions:
- **`text`** — shown verbatim to the user as retrieved evidence.
- **`embed_text`** — text fed to MiniLM: prepends the `scene_summary` (modern English)
  so contemporary-language queries align with archaic dialogue. Truncated to 256 tokens.
- **`scene_id`** — composite key `{play}_{act}_{scene}` used for Parent Document Retrieval.

### `build_speaker_turn_chunks.py`
Merges consecutive utterances from the same speaker within the same scene into a single
"speaker turn". This is the primary chunking strategy: one chunk = one speaker turn.
Rationale: speaker turns are semantically self-contained and map naturally to the
assignment's required metadata schema (play / act / scene / speaker / utterance).

### `build_index.py`
Full indexing pipeline:
1. Loads all speaker-turn records via `data_loader`.
2. Creates chunks via `chunking.create_chunks`.
3. Filters out stage directions and chunks shorter than 4 words.
4. Applies **sliding-window context**: each chunk's `embed_text` is enriched with its
   same-scene predecessor and successor, improving retrieval of context-dependent turns.
5. Builds a `scene_lookup` mapping `scene_id → full scene text` from raw JSON.
6. Encodes all chunks with MiniLM and saves `embeddings.npy` + `chunks_meta.json`.

### `retrieval.py`
`EmbeddingRetriever` encodes a query with MiniLM, computes cosine similarity against
the prebuilt embedding matrix (scikit-learn), and returns the top-k chunks with scores.
Simple and auditable; can be swapped for FAISS or Chroma with no changes to callers.

### `rag_chatbot.py`
Main RAG entry point. `RAGSystem.answer(query)`:
1. Retrieves top-8 chunks.
2. Checks minimum score threshold (0.25) — returns an "out of scope" message for
   unrelated queries.
3. Fetches full scene text from `scene_lookup` for each retrieved chunk (**Parent
   Document Retrieval**), providing richer scene context without inflating chunk size.
4. Builds a structured prompt: scene context → ranked passages with source IDs → question.
5. Generates with Qwen2.5-1.5B-Instruct (greedy, 300 new tokens).
6. Applies citation guard: prefixes stylised responses with
   `[Creative output — not textual evidence]`.

### `baseline.py`
Prompt-only comparison system. Same model, simpler system prompt, **no retrieval**.
Used to demonstrate the improvement from RAG.

### `evaluate.py`
Loads question JSON files, runs every question through both RAG and Baseline, and writes
a pre-filled CSV. Score columns (`correctness`, `grounding`, `retrieval_relevance`,
`usefulness`, `style_quality`) are intentionally left blank for manual annotation.

## Setup

### Step 1 — Install dependencies

```bash
pip install -r requirements.txt
```

### Step 2 — (Optional) Rebuild speaker-turn chunks from utterances

Only needed if you modify the chunking strategy or start from fresh utterance files.

```bash
python3 src/build_speaker_turn_chunks.py
```

### Step 3 — Build the retrieval index

Run once before the first chatbot session. Output files are already committed, so
this step is not required during assessment unless you rebuild from scratch.

```bash
python3 src/build_index.py
```

Writes to `data/processed/`:

| File | Description |
|---|---|
| `chunks.jsonl` | All speaker-turn chunks |
| `embeddings.npy` | MiniLM embedding matrix |
| `chunks_meta.json` | Chunk metadata (parallel to embeddings) |
| `scene_lookup.json` | Full scene text per scene_id (Parent Document Retrieval) |

### Step 4 — Language model

The system uses **Qwen2.5-1.5B-Instruct** (~3.1 GB). On first run the model downloads
automatically from HuggingFace and is cached at `~/.cache/huggingface/`.

To use a local copy instead, place the model weights at:

```
model/qwen2.5-1.5b-instruct/
```

The system detects and uses the local copy automatically if it exists.

### RAG chatbot — interactive mode

python3 rag_chatbot.py

the generated answer.

```bash
cd src

```

### Stylised generation

Ask any query that contains a stylised-generation keyword:

```bash
python3 src/rag_chatbot.py --query "Write a Shakespearean-style response from Juliet explaining her conflict after meeting Romeo."
```

## Design Decisions

### Chunking strategy — speaker turns
Speaker turns (consecutive utterances by the same speaker within a scene) were chosen
over fixed-size windows because they preserve semantic coherence. A character's argument,
soliloquy, or response is kept intact. Each chunk retains full metadata (play / act /
scene / speaker), satisfying the assignment's evidence-display requirement.

### Embedding model — all-MiniLM-L6-v2
A 22 M-parameter sentence transformer. Fast on CPU, fits the assignment's laptop-grade
compute constraint, and produces strong sentence-level semantic representations. The
`embed_text` field prepends the modern-English `scene_summary` to improve alignment
between contemporary queries and archaic Elizabethan text.

### Sliding-window context in embed_text
Each chunk's embedding incorporates its same-scene predecessor and successor, so a
short utterance that is only meaningful in context (e.g., a one-line reply) is
retrievable via its neighbours' content.

### Parent Document Retrieval (PDR)
Speaker-turn chunks are kept small for precise retrieval; full scene text is injected
into the prompt separately via `scene_lookup.json`. This gives the language model
broader context without expanding chunk size or degrading retrieval precision.

### Language model — Qwen2.5-1.5B-Instruct
A 1.5 B-parameter instruction-tuned model. Runs on a standard laptop CPU (no GPU
required). Instruction tuning enables reliable prompt following (citation format,
style guard, out-of-scope detection).

### Out-of-scope detection
If the top retrieval score is below 0.25, the system returns a fixed "out of scope"
message rather than hallucinating. This was validated with the group question Q6
("Who is the president of Australia?").

## Troubleshooting

**`FileNotFoundError: Retrieval index not found`**  
Run `python3 src/build_index.py` to build the index.

**`ModuleNotFoundError`**  
Run `pip install -r requirements.txt`.

**Slow first run**  
The model downloads ~3.1 GB from HuggingFace on the first run. Subsequent runs load
from the local cache.

**MPS / CUDA out of memory**  
Set `PYTORCH_MPS_HIGH_WATERMARK_RATIO=0.0` (macOS) or reduce `max_new_tokens` in
`rag_chatbot.py` and `baseline.py`.
